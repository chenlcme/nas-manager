# Product Requirements Document - nas-manager

**Author:** 立子
**Date:** 2026-04-15

## Executive Summary

**极客工具箱** — 一款面向技术爱好者的 NAS 管理工具箱，采用极简主义设计哲学，帮助用户高效管理音乐文件（查看/编辑/试听验证）并支持多云对象存储备份。

**目标用户：** 有一定技术能力的个人用户，拥有 NAS 设备，追求高效、简洁、开源的工具。

**解决的问题：**
- NAS 音乐文件散乱，元数据不全，批量管理困难
- 公有云存储备份依赖单一厂商，存在锁定风险
- 现有工具臃肿复杂，功能冗余

### What Makes This Special

**播放即验证，现场编辑：** 内置轻量播放器用于验证文件解析结果（标签/封面/歌词），播放时可直接编辑，不做流媒体、不做音乐库，精准服务于元数据管理场景。

**多云防锁定：** 以 S3 协议为兜底，名单内云（阿里云OSS/腾讯云COS等）单独优化，用户不被单一厂商绑定。

**极简部署哲学：** Go 后端内嵌 Preact 前端，编译为单一二进制文件，无需 Docker 即可运行，支持 ARM 架构，拿来就用。

**二期演进清晰：** 一期音乐管理 → 二期多云备份 → 三期照片/视频/下载器路线图已规划。

## Project Classification

- **项目类型:** Web App（Go 内嵌前端，单二进制部署）
- **领域:** 通用工具 / 文件管理
- **复杂度:** Low-Medium
- **项目背景:** Greenfield（全新项目）

## Success Criteria

### User Success

**核心成功时刻：** 用户能快速整理散落的音乐文件，批量修正元数据，通过播放现场验证编辑结果，满意地完成音乐库整理。

- 1000首音乐批量编辑 ≤ 10分钟
- 编辑后验证覆盖率 100%（每首都可播放验证）

**情感指标：** 用户感到"终于有一个工具能理解我的痛点了"——不是播放器，是整理神器。

### Business Success

- 纯粹个人开源项目，无商业目标
- GitHub star 数量作为社区认可度指标（非硬性KPI）

### Technical Success

- 单一 Go 二进制文件，无需 Docker 即可运行
- ARM 架构支持（NAS 常见 ARM 设备）
- SQLite 加密存储凭证，敏感数据不泄漏
- 启动时间 ≤ 3秒
- 支持的音乐格式覆盖率 ≥ 90%（按格式种类计算，MP3/FLAC/APE/OGG等主流格式）

### Measurable Outcomes

| 指标 | 目标 |
|------|------|
| 1000首音乐整理时间 | ≤ 10分钟 |
| 解析正确率 | ≥ 95% |
| 启动时间 | ≤ 3秒 |
| 二进制文件大小 | ≤ 50MB |

## Product Scope

### MVP - Minimum Viable Product

**一期音乐管理核心：**
- 音乐文件浏览（按歌手/专辑/文件夹组织）
- ID3/封面/歌词查看 + 轻量播放器（点一首播一首）
- 播放中直接编辑标签/封面/歌词
- 批量元数据编辑（支持在线歌词搜索）
- 基础搜索（文件名/标签）
- 文件删除（同时删除文件和数据库记录）

### Growth Features (Post-MVP)

- 重复文件检测（Hash去重）
- AI 元数据自动补全（缺失信息从网上抓取）
- 歌词翻译（AI大模型）
- 响应式 UI 优化（PC列表视图 / 手机卡片视图）

### Vision (Future)

- **二期：** 多云对象存储备份（S3协议 + 阿里云OSS/腾讯云COS）
- **三期：** 照片/视频管理 + 下载器

## User Journeys

### 旅程1：首次配置与音乐扫描

**人物：** 小明，NAS 用户

**场景：** 小明在群晖上用 Docker 部署了极客工具箱，挂载了音乐目录 `/music`，指定 SQLite 存储在 `/data/geek-toolbox.db`。

**旅程：**
1. 小明打开 Web 页面，设置音乐目录为 `/music`
2. 点击"扫描"，工具箱遍历目录，解析所有音乐文件的 ID3 标签
3. 扫描完成，展示音乐库：按歌手/专辑组织，共 3,247 首歌曲
4. 小明感叹："不需要做任何配置，直接就能用"

**异常场景：扫描失败**
1. 小明点击"扫描"，工具箱开始遍历目录
2. 扫描过程中遇到 3 个损坏的音频文件，1 个权限不足的文件
3. 系统记录错误，继续扫描其余文件
4. 扫描完成后，展示扫描报告：
   - 成功：3,243 首
   - 失败：4 个文件（2 个损坏，1 个权限不足，1 个格式不支持）
5. 小明点击失败文件，查看具体错误原因
6. 小明可选择重新扫描（跳过失败文件）或手动处理

### 旅程2：批量编辑元数据

**人物：** 小明

**场景：** 一批音乐封面和歌词缺失，小明想批量补全。

**旅程：**
1. 小明选中 50 首缺失封面的歌曲
2. 打开批量编辑，按空格/连字符分隔的词片段在线搜索歌词，搜索失败可手动重试
3. 预览搜索结果，确认后批量应用
4. 用播放器抽查几张，确认正确

### 旅程3：播放并现场编辑

**人物：** 小明

**场景：** 小明想验证某首歌的元数据是否正确，不对的话直接改。

**旅程：**
1. 小明选中一首歌，点击播放
2. 播放器展示封面、歌词、时间信息
3. 小明发现封面不对，点击编辑，直接在播放器面板修改
4. 修改保存后，刷新展示

### 旅程4：配置系统

**人物：** 小明

**场景：** 小明想配置云存储凭证为二期备份做准备。

**旅程：**
1. 小明打开设置，添加阿里云OSS凭证
2. 设置加密密码，凭证加密存储在 SQLite 中
3. 凭证配置完成，等待二期功能开放

### Journey Requirements Summary

- **音乐扫描：** 用户指定目录，工具箱自动遍历解析，存储元数据到 SQLite
- **批量编辑：** 支持批量修改标签/封面/歌词，在线歌词搜索
- **播放编辑：** 轻量播放器展示封面/歌词/时间，播放中可直接编辑
- **系统设置：** 加密密码保护凭证存储

## Web App Specific Requirements

### 技术架构

- **前端架构：** Preact 组件化架构
  - Preact 处理所有交互组件（播放器、批量编辑弹窗）
  - 组件状态驱动 UI 更新
  - SPA-like 体验

- **浏览器支持：** 仅 Chrome，使用通用 Web 规范开发

- **SEO：** 不需要（本地部署工具，无公开搜索需求）

- **实时性：** 不需要 WebSocket，用户操作触发 Preact 状态更新

- **无障碍：** 不需要额外无障碍优化

### 响应式设计策略

| 设备 | 视图模式 |
|------|---------|
| PC | 列表视图 |
| 手机/平板 | 卡片视图 |

- 流式布局，适配不同屏幕宽度
- 播放器组件在任何视图下均可悬浮/固定


## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP 策略：** Problem-Solving MVP — 解决核心痛点（批量整理音乐元数据），提供最小可用工具集

**MVP 团队需求：** 单人开发（立子），Go + Preact 技术栈

**MVP 判断标准：** 用户不能用它高效整理音乐元数据，工具就不算有用

### MVP Feature Set (Phase 1)

**核心用户旅程：**
- 首次配置与音乐扫描（旅程1）
- 批量编辑元数据（旅程2）
- 播放并现场编辑（旅程3）

**必须功能：**
- 音乐目录扫描 + ID3 解析入库
- 按歌手/专辑/文件夹浏览
- 轻量播放器（封面/歌词/时间展示，点一首播一首）
- 播放中直接编辑标签/封面/歌词
- 批量元数据编辑（支持在线歌词搜索）
- 基础搜索（文件名/标签）
- 文件删除（同时删除文件和数据库记录）

### Post-MVP Features

**Phase 2（Growth）：**
- 重复文件检测（Hash去重）
- AI 元数据自动补全
- 歌词翻译
- 响应式 UI 优化

**Phase 3（Expansion）：**
- 多云对象存储备份（S3 + 阿里云OSS/腾讯云COS）
- 照片/视频管理
- 下载器

### Risk Mitigation Strategy

**技术风险：**
- ID3 解析库成熟，风险低
- Go 内嵌前端有成熟方案（embed），风险低
- 在线歌词搜索依赖第三方API，需考虑降级策略

**市场风险：**
- 开源项目无商业风险
- 社区反馈迭代开发

**资源风险：**
- 单人开发，降低协调成本
- MVP 功能范围可控

## Functional Requirements

### 1. 音乐扫描与导入

- FR1: 用户可以指定音乐目录路径
- FR2: 系统可以遍历指定目录，识别音乐文件（支持 MP3/FLAC/APE/OGG 等主流格式）
- FR3: 系统可以解析音乐文件的 ID3 标签（标题/艺术家/专辑/年份/曲目号等）
- FR4: 系统可以将解析后的元数据存储到 SQLite 数据库
- FR5: 用户可以触发重新扫描（支持增量扫描和全量扫描，增量扫描依据文件创建时间和最后修改时间判定）
  - **验收标准：** Given 已存在的扫描结果，When 用户触发增量扫描，Then 仅重新解析修改时间晚于上次扫描时间的文件；增量扫描耗时 < 全量扫描的 10%
- FR6: 系统可以检测并清理孤岛记录（文件已删除但数据库仍有记录）
- FR7: 系统可以处理扫描异常（文件损坏、权限不足、网络超时），并向用户报告
  - **验收标准：** Given 扫描过程中遇到异常文件，When 异常发生，Then 系统记录错误日志并继续扫描，返回扫描报告列出所有失败文件及原因

### 2. 音乐浏览与组织

- FR8: 用户可以按歌手视图浏览音乐库
- FR9: 用户可以按专辑视图浏览音乐库
- FR10: 用户可以按文件夹结构浏览音乐库
- FR11: 用户可以在浏览视图中按名称/时长/添加时间排序
- FR12: 用户可以查看单曲的详细信息（ID3标签/封面/歌词）
- FR13: 用户可以在列表中多选多个音乐文件
- FR14: 用户可以删除选中的音乐文件（同时删除文件和数据库记录）
  - **验收标准：** Given 用户选中 N 首歌曲并点击删除，When 确认删除，Then 系统同时删除磁盘文件和数据库记录，返回删除结果报告

### 3. 播放器与现场编辑

- FR15: 用户可以播放选中的音乐文件（点一首播一首，无播放队列）
- FR16: 播放器可以展示专辑封面图片
- FR17: 播放器可以展示歌词（同步或静态）
- FR18: 播放器可以展示播放时间信息
- FR19: 用户在播放过程中可以直接编辑当前歌曲的标签/封面/歌词
  - **验收标准：** Given 歌曲正在播放，When 用户点击编辑并修改标签/封面/歌词，Then 修改保存后播放器面板刷新展示新内容，数据库同步更新

### 4. 批量编辑与元数据补全

- FR20: 用户可以批量修改选中音乐的标签（艺术家/专辑/标题等）
- FR21: 用户可以批量修改选中音乐的封面图片
- FR22: 用户可以搜索并批量应用歌词（系统按文件名分隔的词片段在线搜索歌词，搜索失败时用户可手动重试，API来源在实现时确定）
  - **验收标准：** Given 用户选中多首无歌词歌曲并触发歌词搜索，When 搜索完成，Then 显示匹配结果列表，用户确认后批量写入歌词；搜索失败时显示重试按钮
- FR23: 系统支持依次撤销批量编辑操作（每次撤销一个批量记录）
  - **验收标准：** Given 用户执行过多次批量编辑，When 用户触发撤销，Then 系统恢复最近一次批量编辑之前的状态，重复撤销可逐步恢复

### 5. 搜索

- FR24: 用户可以按文件名搜索音乐
- FR25: 用户可以按 ID3 标签内容搜索音乐

### 6. 系统设置

- FR26: 用户可以设置加密密码，系统基于密码生成加密密钥
- FR27: 系统可以在首次访问时引导用户完成基础配置（设置音乐目录和SQLite存储路径两步完成）

## Non-Functional Requirements

### Performance

| 指标 | 要求 |
|------|------|
| 启动时间 | ≤ 3秒 |
| 二进制文件大小 | ≤ 50MB |
| UI 操作响应 | ≤ 200ms |
| 首屏加载（扫描结果展示） | ≤ 2秒 |
| 1000首音乐扫描入库 | ≤ 10分钟 |

### Security

| 需求 | 说明 |
|------|------|
| 凭证加密 | 云存储凭证使用用户密码生成的密钥加密存储 |
| 密钥管理 | 密钥由用户密码派生，丢失后凭证无法解密（用户需重新配置凭证） |

### Scalability

- 单用户本地部署，无并发用户需求
- SQLite 存储，本地 NAS 性能足够

### Platform

| 需求 | 说明 |
|------|------|
| 架构支持 | AMD64 + ARM64 |
| 部署方式 | 单一 Go 二进制文件，直接运行 |
| 浏览器支持 | Chrome（桌面端 + 移动端） |
